# Sprint A04 — Part 1 / Claude Code (Cursor) Prompt — Revit Sensor v0.2

**What this is:** upgrades the existing `legacy-revit-sensor` C# add-in from v0.1 to v0.2. After a snapshot is created, the add-in exports every sheet in the model as PNG (two resolutions: 512px thumbnail + 2048px full) and uploads them to the new `/api/snapshots/:id/sheets` multipart endpoint with progress UI in Revit.

**This is part 1 of Sprint A04.** Run this in Cursor + Claude Code FIRST. Then run part 2 (Replit Agent prompt) which adds the server endpoint, storage, viewer, and Claude Vision integration.

**Prerequisites:**
- Sprint A03 verified complete on Replit ✓
- Existing `legacy-revit-sensor` v0.1 working at `c:\legacy-revit-sensor\LegacyRevitSensor\` (or wherever you put it)
- Revit 2026 installed
- .NET 8 SDK installed

**How to use:**
1. Open Cursor with the legacy-revit-sensor folder as workspace root (verify path in status bar)
2. Open Claude Code panel
3. Paste everything between BEGIN and END markers below

**After it builds:**
- Close all Revit instances, reopen Revit
- Open Snowdon Towers Sample (or any model)
- Add-Ins → Design Tools → Send Snapshot
- Watch the new progress dialog: "Capturing snapshot... → Exporting sheet 1/55... → Uploading sheets..."
- Open Replit Workbench → snapshot appears as before; sheets endpoint receives uploads (server-side handling comes in part 2)

SECTION 5 — Snapshot/SheetExporter.cs (NEW)
============================================================

public static class SheetExporter

Responsibilities:
1. Enumerate all ViewSheets in the document (FilteredElementCollector).
2. For each sheet, export to PNG twice — once at 512px wide (thumbnail), once at 2048px wide (full).
3. Use Revit's ImageExportOptions with ImageFileType.PNG, ZoomType.FitToPage, FitDirection.Horizontal, PixelSize set per pass.
4. Read each exported PNG file from disk into a byte[], then delete the temp file.
5. Cap at 200 sheets per snapshot to prevent runaway exports — if over, take first 200 sorted by SheetNumber and add a warning.
6. Provide progress callback so the UI can update.

Public API:

  public static IReadOnlyList<SheetUploadEntry> ExportAllSheets(
      Document doc,
      IProgress<SheetExportProgress>? progress = null,
      CancellationToken cancellationToken = default
  )

  public record SheetExportProgress(int Current, int Total, string CurrentSheetLabel);

Implementation notes:

- ImageExportOptions requires a temp file path with .png extension. Use `Path.GetTempFileName()` then change extension to .png, OR use Path.Combine(Path.GetTempPath(), Guid.NewGuid() + ".png"). Latter is safer; first sometimes leaves zero-byte temp files.
- Set options.FilePath to that temp path.
- Set options.ImageResolution = ImageResolution.DPI_150 for thumbnail pass, DPI_300 for full pass (Revit's DPI options are an enum: DPI_72, DPI_150, DPI_300, DPI_600).
- Actually — DPI is independent of pixel size. Use options.PixelSize = 512 (thumbnail) or 2048 (full). PixelSize is the longest edge in pixels.
- Set options.ZoomType = ZoomType.FitToPage and options.FitDirection = FitDirection.Horizontal.
- Set options.HLRandWFViewsFileType = ImageFileType.PNG.
- Set options.ShadowViewsFileType = ImageFileType.PNG.
- IMPORTANT: ImageExportOptions wants a SET of view IDs. Use options.SetViewsAndSheets(new List<ElementId> { sheet.Id }).
- Call doc.ExportImage(options).
- Revit appends the view name to the filename. The actual output path becomes "{tempPath without .png} - Sheet - {sheetNumber} - {sheetName}.png" or similar — Revit's naming is annoyingly variable. Strategy: use a unique temp DIRECTORY per export (Path.Combine(Path.GetTempPath(), "lrs_" + Guid.NewGuid())), set FilePath to a base name in that dir, then after ExportImage call Directory.GetFiles on the dir to find what Revit actually wrote (there should be exactly one .png), read it, delete the file and the directory.
- Read pixel dimensions from the PNG before discarding (use System.Drawing.Image.FromFile or BitmapDecoder, capture width/height into the SheetUploadEntry).
- For revision data: each sheet has GetCurrentRevision() returning an ElementId; if non-invalid, look up the Revision and get its RevisionNumber + RevisionDate; otherwise null.

Error handling:
- Wrap each sheet in try/catch. If a single sheet fails, log to stderr, continue. Add to a "failed sheets" count.
- Return all successfully exported sheets even if some failed.

Progress reporting:
- Each sheet export = 2 work units (thumbnail + full). Total = sheets.Count * 2.
- Report after each pass: current="{passLabel} {sheet.SheetNumber} - {sheet.Name}".

Cancellation:
- Check cancellationToken.ThrowIfCancellationRequested() between sheets.

SECTION 6 — Http/SheetUploadClient.cs (NEW)
============================================================

public static class SheetUploadClient

Public API:

  public static async Task<SheetUploadResponse> UploadAsync(
      string snapshotId,
      IReadOnlyList<SheetUploadEntry> sheets,
      Settings settings,
      IProgress<UploadProgress>? progress = null,
      CancellationToken cancellationToken = default
  )

  public record UploadProgress(int Current, int Total, string CurrentSheetLabel);

Implementation:

POST {settings.ReplitUrl}/api/snapshots/{snapshotId}/sheets

Use multipart/form-data with one PART per file. Naming convention:
- For each sheet i: 
  - field name "sheet_{i}_thumb", filename "thumb_{sheetNumber}.png", content-type image/png, body = sheet.ThumbnailPng
  - field name "sheet_{i}_full", filename "full_{sheetNumber}.png", content-type image/png, body = sheet.FullPng
- Plus a single "metadata" field, content-type application/json, body = JSON-serialized array of:
  [{ "index": i, "sheetNumber": ..., "sheetName": ..., "viewCount": ..., "revisionNumber": ..., "revisionDate": ..., "thumbnailWidth": ..., "thumbnailHeight": ..., "fullWidth": ..., "fullHeight": ... }, ...]

Use System.Net.Http.MultipartFormDataContent. Add ByteArrayContent for each file part with explicit ContentType ("image/png") and StringContent for metadata with ContentType "application/json".

Required header: x-snapshot-secret: settings.Secret (same auth as snapshot POST)

Timeout: 5 minutes (sheets can be large + slow Replit instances).

Streaming/buffering: this is fine to fully buffer in memory for prototype (Snowdon's 55 sheets at ~28MB total is well under 100MB; HttpClient can handle it).

Progress reporting: report progress as parts are added to the multipart, not as bytes are uploaded — HttpClient doesn't expose byte-level upload progress without a custom HttpContent wrapper. Per-sheet granularity is fine for v0.2.

Error handling: on non-2xx status, throw HttpRequestException with the response body and status code. Don't swallow.

Response parsing: read body, deserialize as SheetUploadResponse JSON (camelCase). Pass through to caller.

SECTION 8 — MODIFY Commands/SendSnapshotCommand.cs
============================================================

Restructure Execute() to:

1. Validate settings (existing). Show TaskDialog if not configured (existing).
2. Get UIDocument (existing). Show TaskDialog if no active document (existing).
3. Build snapshot JSON (existing — call SnapshotBuilder.Build).
4. Open ProgressDialog (NEW). Set status "Capturing snapshot...". Show non-modally on Revit's main window.
5. Run upload in a Task — but Revit's API is single-threaded and ImageExport must run on the API thread. Approach:
   - Step A (UI thread): Snapshot POST first via SnapshotClient.PostAsync (this is HTTP only, can run in UI thread synchronously; it's fast)
   - Step A on success: progressDialog.SetStatus("Snapshot received. Exporting sheets...");
   - Step B (UI thread): SheetExporter.ExportAllSheets(doc, progress, token) — runs synchronously on the Revit API thread. This blocks the dialog's UI updates because the Revit thread is busy. Use Application.DoEvents() after each sheet to keep the dialog responsive (acceptable for v0.2; not ideal but works).
   - Step C (background Task): pass exported sheets to SheetUploadClient.UploadAsync. This is HTTP only and can be on a worker thread.
6. After Step C completes: progressDialog.MarkComplete("Done — {n} sheets uploaded").
7. Show success TaskDialog with both snapshot id AND sheet upload counts (uploaded / skipped / failed).
8. If user cancels via ProgressDialog: cancel the export OR upload (whichever phase is active). Show cancellation TaskDialog.
9. If any phase fails:
   - Snapshot fails: show error, return Result.Failed (don't attempt sheet upload).
   - Sheet export fails partially: still upload what succeeded, show success with warning.
   - Sheet upload fails: snapshot is still saved on the server (it succeeded in Step A); show TaskDialog with the snapshot id and the upload error, and let the user know they can retry by clicking Send Snapshot again (idempotency on the server side handles this gracefully — see part 2).

Important behavior preservation:
- v0.1 path of "snapshot only, no sheets" still works if SheetExporter throws unexpectedly. Snapshot was already POSTed in Step A. The user gets a warning but the snapshot is real.
- The Configure command is unchanged.

SECTION 12 — DO NOT BUILD
============================================================

- Smart delta export (only changed sheets) — explicitly out of scope per Sprint A04 strategy decision; will be a future optimization sprint
- Sheet content hashing — same reason
- A new ribbon button for "sheets only" — the v0.1 Send Snapshot button does both phases now
- Background-thread Revit API calls (ImageExportOptions must run on the API thread)
- Server-side endpoint /api/snapshots/{id}/sheets — that's part 2's job; this prompt only POSTs to it
- Compression of PNGs beyond what Revit's PNG export does natively (it's already PNG; gzipping the multipart adds CPU and gets us ~20% — not worth it at this scale)
- Resumable / chunked upload — single multipart POST is fine for prototype
- Encryption beyond HTTPS (which Replit gives us by default)
- A way to disable sheet export per-snapshot — keep it simple; if architects don't want sheets they can use a separate model
- Parallel export of multiple sheets (Revit API isn't thread-safe)
- Vector PDF export instead of PNG — PDF is the eventual right answer for vector quality but PNG is what Vision-capable LLMs consume today
