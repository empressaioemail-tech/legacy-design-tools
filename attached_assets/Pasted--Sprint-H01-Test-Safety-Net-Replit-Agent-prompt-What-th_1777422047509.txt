# Sprint H01 — Test Safety Net (Replit Agent prompt)

**What this is:** Sprint H01 establishes a minimum viable test safety net so Sprint A06 (AI Reviewer reasoning loop) can ship structural changes without silently breaking the working A04+A05 pipeline.

**Run this AFTER:** Sprint A05 verified ✓ (citation chips working on real Moab engagement, 264 atoms embedded, embedding backfill complete).

**Stack:** Vitest (already in the Vite ecosystem, zero new deps to choose), GitHub Actions on push/PR, practical strictness (critical paths get tests, peripheral code is best-effort).

**Two parts.** Part 1 is test infrastructure + lib unit tests. Part 2 is api-server integration + frontend components + CI workflow. Run Part 1 to completion before Part 2.

**This is a multi-day sprint.** Estimated 7-10 days of agent work split across both parts. Don't try to one-shot.

**How to use:** paste Part 1 first into Replit Agent. After Part 1 verifies, paste Part 2.

SECTION 2 — TEST FIXTURE CONVENTIONS
============================================================

Establish conventions so future agents (and humans) follow consistent patterns:

  Source code:           lib/<package>/src/foo.ts
  Unit tests:            lib/<package>/src/foo.test.ts (colocated)
  Integration tests:     lib/<package>/src/__tests__/integration/foo.integration.test.ts
  Fixtures (static data) lib/<package>/src/__fixtures__/foo.html (or .pdf, .json, etc.)
  Test utilities:        lib/<package>/src/__tests__/utils.ts

Fixtures are real-world data captured at a point in time:
  - HTML fixtures from Grand County's actual page (saved as static .html)
  - PDF text fixtures from the IWUIC PDF (saved as .txt of extracted text, since PDF parsing is tested separately)
  - Municode API JSON fixtures from real responses (saved as .json)

Why fixtures and not live calls in tests:
  - Tests run offline, deterministic, fast
  - Source layout changes are caught (test fails when fixture doesn't match parser expectations)
  - No API costs in tests, no Municode rate limit concerns
  - Fixtures get refreshed manually when source changes (rare)
