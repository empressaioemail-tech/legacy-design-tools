    fullPng         bytea not null
    fullWidth       integer not null
    fullHeight      integer not null
    sortOrder       integer not null           (Revit's natural sheet order)
    createdAt       timestamptz default now() not null
  
  indexes:
    on (snapshotId, sortOrder) — for paginated grid display
    on (engagementId, sheetNumber) — for cross-snapshot lookup
    on (snapshotId, sheetNumber) UNIQUE — prevent duplicate sheets within a snapshot

  POST /snapshots/{id}/sheets
    Auth: header x-snapshot-secret (same as snapshot POST)
    Request: multipart/form-data
      - field "metadata": JSON-encoded SheetMetadata[] (described below)
      - fields "sheet_{i}_thumb": image/png (per sheet i)
      - fields "sheet_{i}_full": image/png (per sheet i)

    Response: SheetUploadResponse
    Behavior: idempotent on (snapshotId, sheetNumber) — re-uploading replaces existing record (same sheet number gets updated PNG bytes); ON CONFLICT (snapshotId, sheetNumber) DO UPDATE.
    Error responses: 401 (bad secret), 404 (snapshot not found), 413 (payload too large; cap at 100MB total request size for safety), 415 (wrong content-type)

  GET /snapshots/{id}/sheets
    Returns SheetSummary[] for the snapshot, sorted by sortOrder.

  GET /sheets/{id}/thumbnail.png
    Returns image/png response with Cache-Control: max-age=86400 (24h — they're immutable once uploaded). Body is the thumbnailPng bytea.

  GET /sheets/{id}/full.png
    Same pattern, returns fullPng bytea, Cache-Control: max-age=86400.

EXTEND existing endpoint:

SECTION 3 — BACKEND: MULTIPART INGESTION (artifacts/api-server)
============================================================

Add multipart parsing dependency: busboy (lightweight, streaming, well-supported with Express). NOT multer (older API, less control).

Add to artifacts/api-server: `npm install busboy` (or pnpm add).

New route: artifacts/api-server/src/routes/sheets.ts

Implements:

  POST /api/snapshots/:snapshotId/sheets

Logic:
  1. Validate x-snapshot-secret → 401 if mismatch
  2. Validate snapshotId exists in DB → 404 if not, capture engagementId
  3. Stream-parse the multipart body with busboy
  4. As fields/files arrive:
     - "metadata" field → buffer, JSON.parse into MetadataEntry[]
     - "sheet_{i}_thumb" file → buffer to memory (cap 5MB per file), index by `i`
     - "sheet_{i}_full" file → buffer to memory (cap 20MB per file), index by `i`
  5. After busboy 'finish' event:
     - Validate every metadata entry has matching thumb + full files (skip with error otherwise)
     - For each entry, prepare a row with snapshotId, engagementId, sheet fields, png bytea data, sortOrder = metadata.index
     - INSERT in a single transaction with ON CONFLICT (snapshotId, sheetNumber) DO UPDATE SET (all fields except id, createdAt) — idempotent re-upload
  6. Update snapshots table: set a sheetCount column on the snapshot row (denormalized count)
     - Add sheetCount column to snapshots table if not present (additive migration in this prompt's schema work)
  7. Return SheetUploadResponse: { uploaded, skipped, failed, errors }

Validation:
- Total request body size cap: 150MB (Express default is 100KB without explicit limit; raise this for the multipart route). Express config: app.use("/api/snapshots/:id/sheets", ...) with raw body size limit.
- Per-file size caps enforced inside busboy (set "limits.fileSize" option).
- Reject non-PNG content-type on the file fields.
- Treat any individual sheet failure as: skip that sheet, add error message to errors array, continue. Only fail the whole request on schema-level errors (no metadata, no auth, etc.).

Error handling:
- If busboy emits 'error' or limits exceeded, respond 413 with a clear message.
- If DB INSERT fails, respond 500 with snapshot id (so the user can retry).

GET endpoints:

  GET /api/snapshots/:id/sheets
    SELECT sheets WHERE snapshotId = :id ORDER BY sortOrder ASC
    Return SheetSummary[] (NO png data — just metadata)

  GET /api/sheets/:id/thumbnail.png
    SELECT thumbnailPng FROM sheets WHERE id = :id
    Set Content-Type: image/png
    Set Cache-Control: max-age=86400, immutable
    Set ETag: hash of the bytes (or just the sheet id since they're immutable)
    Return the bytea as response body.
    404 if no row.

  GET /api/sheets/:id/full.png
    Same pattern with fullPng.

Update existing routes/snapshots.ts:
  - GET /api/snapshots/:id detail response now includes the sheets array (SheetSummary, no PNGs)
  - GET /api/snapshots flat list response: each snapshot now includes sheetCount

Update existing routes/engagements.ts:
  - GET /api/engagements: each engagement's latestSnapshot includes sheetCount
  - GET /api/engagements/:id: each snapshot in the snapshots array includes sheetCount

============================================================

Update POST /api/chat:

Body now accepts: { engagementId, question, history?, referencedSheetIds?: string[] }

Logic:
  1. Existing logic stays (look up engagement, latest snapshot, build system prompt with snapshot JSON).
  2. NEW: if referencedSheetIds is non-empty:
     - SELECT sheets WHERE id IN (referencedSheetIds) AND engagementId = :engagementId (security: prevent cross-engagement access)

       { type: "image", source: { type: "base64", media_type: "image/png", data: base64-encoded fullPng } }
     - Augment the user message to include both:
       - Text: "User question: " + question + "\n\nThe following sheets are attached for visual reference: " + sheets.map(s => `${s.sheetNumber} ${s.sheetName}`).join(", ")
       - Image content blocks (one per sheet)
     - Cap at 4 sheets per request to stay within Claude's image content block limits and reasonable token budgets.
  3. Stream Claude response via existing @workspace/integrations-anthropic-ai (model claude-sonnet-4-6, max_tokens unchanged, stream true)

Important: when calling the Anthropic SDK (via the integrations package), the messages array now needs to support content blocks (an array) not just plain string content. If the integrations package only exposes plain-text helpers, extend it minimally to accept structured content blocks for the user message.
